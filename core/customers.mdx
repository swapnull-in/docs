## Overview

Customers represent the people or companies who pay you. Each customer can
carry multiple subscriptions, a card on file, [prepaid wallets](/advanced/wallets),
[entitlements](/advanced/entitlements), DPDP [consent records](/compliance/consent-tracking),
and a [churn score](/advanced/churn-prediction).

## Create a Customer

<CodeGroup>
```bash cURL
curl -X POST https://api.recurso.dev/v1/customers \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Acme Corporation",
    "email": "billing@acme.com",
    "phone": "+919876543210",
    "line1": "123 MG Road",
    "city": "Bangalore",
    "state": "Karnataka",
    "zip": "560001",
    "country": "IN"
  }'
```

```typescript Node
const customer = await recurso.customers.create({
  name: 'Acme Corporation',
  email: 'billing@acme.com',
  phone: '+919876543210',
  line1: '123 MG Road',
  city: 'Bangalore',
  state: 'Karnataka',
  zip: '560001',
  country: 'IN',
});
```

```go Go
customer, _ := client.Customers.Create(ctx, &recurso.CustomerCreateParams{
  Name:    "Acme Corporation",
  Email:   "billing@acme.com",
  Country: "IN",
})
```
</CodeGroup>

<Note>
The billing address is **flat** (`line1`, `city`, `state`, `zip`,
`country`), not a nested object. `state` and `zip` matter: they drive
CGST/SGST-vs-IGST for India and the sales-tax lookup for US buyers.
</Note>

## B2B Customers (India)

For business customers in India, capture the GSTIN and tax profile for
proper invoicing:

```bash
curl -X POST https://api.recurso.dev/v1/customers \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "TechCorp India Pvt Ltd",
    "email": "accounts@techcorp.in",
    "gstin": "29ABCDE1234F1Z5",
    "tax_type": "business",
    "place_of_supply": "29",
    "line1": "456 Koramangala",
    "city": "Bangalore",
    "state": "Karnataka",
    "zip": "560034",
    "country": "IN"
  }'
```

<ParamField body="gstin" type="string">
  15-character Indian GST identification number. When present, invoices
  show the buyer's legal name and registration, and B2B e-invoicing rules
  apply.
</ParamField>
<ParamField body="tax_type" type="string">
  `business` or `consumer` — drives B2B vs B2C invoice treatment.
</ParamField>
<ParamField body="place_of_supply" type="string">
  Indian state code (e.g. `29` for Karnataka) — decides CGST/SGST vs IGST.
</ParamField>

<Info>
See the [GST Invoicing guide](/compliance/gst-invoicing) for how these
fields flow onto the tax invoice.
</Info>

## List and Search Customers

<CodeGroup>
```bash cURL
curl -G https://api.recurso.dev/v1/customers \
  -H "Authorization: Bearer $API_KEY" \
  -d search=acme \
  -d country=IN \
  -d status=active \
  -d limit=20 \
  -d page=1
```

```typescript Node
const customers = await recurso.customers.list({
  search: 'acme',
  country: 'IN',
  status: 'active',
});
```
</CodeGroup>

| Parameter | Type | Description |
|-----------|------|-------------|
| `search` | `string` | Match against name and email |
| `country` | `string` | ISO 3166-1 alpha-2 code |
| `status` | `string` | `active` or `inactive` |
| `limit` | `integer` | Results per page (default 20, max 100) |
| `page` | `integer` | 1-based page number |

<Note>
The customer API surface is deliberately small: create, list, and the
card-on-file update below. There is no general update or delete endpoint
today — identity changes and deletion are dashboard/support operations, and
payment details change through the [portal](/portal/overview) flows.
</Note>

## Card on File

`PUT /v1/customers/{id}/payment-method` updates the card **display
metadata** Recurso shows on the customer and dunning surfaces:

```bash
curl -X PUT https://api.recurso.dev/v1/customers/{customer_id}/payment-method \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "card_brand": "visa",
    "card_last4": "4242",
    "card_exp_month": 12,
    "card_exp_year": 2028
  }'
```

<Tip>
The *actual* payment instrument lives at the gateway. Customers change it
themselves through the [Customer Portal](/portal/self-service) — Stripe
SetupIntent or a fresh UPI mandate — or through your checkout. When a
payment fails on an expired card, send a [magic link](/portal/magic-links)
so they can fix it in one click.
</Tip>

## Customer Portal

The portal is customer-facing and passwordless — customers request a
**magic link** by email; no API key or session management on your side:

```bash
# Customer-initiated (or triggered by your app on their behalf)
curl -X POST https://billing.example.com/portal/auth/request \
  -H "Content-Type: application/json" \
  -d '{ "email": "billing@acme.com" }'
```

The emailed link opens a 7-day portal session where customers view and pay
invoices, update payment methods, download PDFs, and cancel subscriptions.
See the [Customer Portal guide](/portal/overview).

## Churn Score

Recurso scores every customer's churn risk from real billing signals
(payment failures, dunning history, subscription age):

```bash
curl https://api.recurso.dev/v1/customers/{customer_id}/churn \
  -H "Authorization: Bearer $API_KEY"
```

```json
{
  "data": {
    "customer_id": "…",
    "score": 73,
    "risk_level": "high",
    "features": { "failed_payments_30d": 2, "…": "…" },
    "model_version": "v1"
  }
}
```

`score` is 0–100; `risk_level` is `low` / `medium` / `high` / `critical`.
See [Churn Prediction](/advanced/churn-prediction) for high-risk lists and
alerts.
