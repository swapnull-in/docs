## Overview

Usage-based billing charges customers based on consumption rather than a fixed price. Common use cases:

- API calls
- Storage (GB)
- Compute hours
- Active users
- Transactions processed

## How It Works

```mermaid
sequenceDiagram
    participant App as Your App
    participant R as Recurso
    participant C as Customer

    App->>R: Report usage events
    Note over R: Aggregates usage
    R->>R: End of billing period
    R->>R: Calculate charges
    R->>C: Generate invoice
```

## Report Usage

Report usage events throughout the billing period. The `POST /v1/usage/events` endpoint accepts the following fields:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `subscription_id` | `string` | Yes | The subscription to report usage for |
| `customer_id` | `string` | No | Customer ID (inferred from subscription if omitted) |
| `dimension` | `string` | No | Usage dimension/meter name (for multi-meter plans) |
| `quantity` | `number` | Yes | Amount of usage to record |
| `action` | `string` | No | `increment` (default) or `set` for gauge metrics |
| `timestamp` | `string` | No | ISO 8601 timestamp (defaults to now) |
| `idempotency_key` | `string` | No | Prevent duplicate events |

<CodeGroup>
```typescript TypeScript
await recurso.usage.record({
  subscription_id: 'sub_xyz',
  quantity: 100,
  timestamp: new Date(),
  action: 'increment',  // or 'set' for gauge metrics
  idempotency_key: 'request_abc123'  // Prevent duplicates
});
```

```bash cURL
curl -X POST https://billing.example.com/v1/usage/events \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "subscription_id": "sub_xyz",
    "customer_id": "cust_abc123",
    "dimension": "api_calls",
    "quantity": 100,
    "action": "increment"
  }'
```
</CodeGroup>

### Batch Reporting

For high-volume usage, batch multiple events:

```typescript
await recurso.usage.recordBatch([
  { subscription_id: 'sub_xyz', quantity: 50, timestamp: t1 },
  { subscription_id: 'sub_xyz', quantity: 75, timestamp: t2 },
  { subscription_id: 'sub_xyz', quantity: 120, timestamp: t3 }
]);
```

## Pricing Models

### Graduated Tiers

Each tier applies to its range:

```typescript

  { up_to: 1000, unit_amount: 0.10 },   // 1-1000: â‚ą100
  { up_to: 5000, unit_amount: 0.05 },   // 1001-5000: â‚ą200
  { up_to: 'inf', unit_amount: 0.01 }   // 5001+: â‚ą0.01 each
]
// Example: 7000 calls = â‚ą100 + â‚ą200 + â‚ą20 = â‚ą320
```

### Volume Tiers

Final tier applies to ALL units:

```typescript

tiers: [
  { up_to: 1000, unit_amount: 0.10 },
  { up_to: 5000, unit_amount: 0.05 },
  { up_to: 'inf', unit_amount: 0.01 }
]
// Example: 7000 calls = 7000 Ă— â‚ą0.01 = â‚ą70
```

### Package Pricing

Charge per bundle:

```typescript

  { up_to: 1000, flat_amount: 999 },    // â‚ą999 for up to 1000
  { up_to: 5000, flat_amount: 1999 },   // â‚ą1999 for up to 5000
```

## Hybrid Billing

Combine a flat base fee with metered usage charges. This is the most common model for SaaS products that want predictable revenue with a usage component.

```typescript
const plan = await recurso.plans.create({
  name: 'Pro + Usage',
  interval: 'month',
  prices: [
    { amount: 4999, currency: 'INR' }  // Base fee: â‚ą49.99/mo
  ],
  usage_prices: [{
    unit_label: 'API call',
    currency: 'INR',
    tiers_mode: 'graduated',
    tiers: [
      { up_to: 10000, unit_amount: 0 },      // 10k calls included
      { up_to: 100000, unit_amount: 0.05 },   // Overage: â‚ą0.05/call
      { up_to: 'inf', unit_amount: 0.01 }
    ]
  }]
});
```

At invoice time, the customer sees two line items:

| Line Item | Amount |
|-----------|--------|
| Pro Plan - Base Fee | â‚ą4,999.00 |
| API Calls - 45,000 calls (10,000 included, 35,000 overage) | â‚ą1,750.00 |
| **Total** | **â‚ą6,749.00** |

<Tip>
Hybrid billing works well for plans where you want a minimum commitment (the base fee) while still charging for heavy usage. Customers who stay under the included usage only pay the flat fee.
</Tip>
