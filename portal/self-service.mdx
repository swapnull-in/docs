## Overview

Once a customer has a portal session ([magic link](/portal/magic-links)),
these `/portal/api/*` actions are available — all authenticated by the
`portal_session` cookie. This is the real, shipped surface; which of these
you expose is configured in the dashboard under **Settings → Portal**.

## View & pay invoices

```bash
# List the customer's invoices
curl https://billing.example.com/portal/api/invoices -b cookies.txt

# Download one as a PDF
curl https://billing.example.com/portal/api/invoices/{invoice_id}/pdf \
  -b cookies.txt -o invoice.pdf
```

To pay, send the customer to the hosted checkout for the open invoice
(`/checkout/{invoice_id}`) — see [Payments](/core/payments).

## Update the card on file (Stripe)

Card updates use a Stripe **SetupIntent** so the card never touches your
servers. Start it, collect the card with Stripe Elements using the returned
`client_secret`, then confirm:

```bash
# 1. Start — returns a client_secret for Stripe Elements
curl -X POST https://billing.example.com/portal/api/payment-method/setup-intent \
  -b cookies.txt

# 2. (browser) confirm the SetupIntent with Stripe.js using client_secret

# 3. Finalize — saves the card metadata on the customer
curl -X POST https://billing.example.com/portal/api/payment-method/confirm \
  -H "Content-Type: application/json" \
  -b cookies.txt \
  -d '{ "setup_intent_id": "seti_..." }'
```

The confirm call returns `status: "saved"` with the new card's brand and
last four, or `status: "processing"` for methods that settle asynchronously.

<Note>
If the deployment isn't configured for self-serve card updates, these
endpoints return a clear "not available on this deployment" error rather
than failing silently.
</Note>

## Re-authorize a UPI AutoPay mandate

For Indian customers on UPI AutoPay, the portal issues a fresh mandate and
returns a hosted authorization URL to redirect to:

```bash
curl -X POST https://billing.example.com/portal/api/payment-method/mandate \
  -H "Content-Type: application/json" \
  -b cookies.txt \
  -d '{ "vpa": "customer@upi" }'    # vpa optional — Razorpay collects it if omitted
```

```json
{ "data": { "auth_url": "https://razorpay.com/..." } }
```

Redirect the customer to `auth_url` to approve the mandate in their UPI app.

## Raise a dispute

A customer can query or dispute one of their own invoices:

```bash
curl -X POST https://billing.example.com/portal/api/invoices/{invoice_id}/dispute \
  -H "Content-Type: application/json" \
  -b cookies.txt \
  -d '{ "reason": "I was charged twice for this invoice." }'
```

Open disputes are listed at `GET /portal/api/disputes`.

## Redeem a gift code

```bash
curl -X POST https://billing.example.com/portal/api/redeem \
  -H "Content-Type: application/json" \
  -b cookies.txt \
  -d '{ "code": "GIFT-XXXX" }'
```

See [Gifts](/advanced/gifts) for how codes are purchased.

## Cancellation

Subscription cancellation runs through the dedicated
[cancellation flow](/portal/cancel-self-service) — surveys, retention
offers, and confirmation — rather than a single portal endpoint.

## Log out

```bash
curl -X POST https://billing.example.com/portal/api/logout -b cookies.txt
```
