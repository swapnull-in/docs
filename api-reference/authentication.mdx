## API Keys

Authenticate all API requests with your secret API key in the `Authorization`
header as a Bearer token:

```bash
curl https://api.recurso.dev/v1/customers \
  -H "Authorization: Bearer $RECURSO_API_KEY"
```

Keys are secret credentials. Newly created keys are prefixed by mode —
`rsk_test_…` or `rsk_live_…`. The demo dataset ships with the fixed test key
`sk_test_12345` for local exploration.

<Warning>
**Never expose API keys in client-side code.** Keys are secret and must only be
used from your backend.
</Warning>

## Test vs. live keys

Every key is either a **test** key (`rsk_test_…`) or a **live** key
(`rsk_live_…`). A new account starts with a test key.

| Type | Prefix | For |
| --- | --- | --- |
| Test | `rsk_test_` | Development and sandbox |
| Live | `rsk_live_` | Production (real money) |

A key's mode must match the server's payment configuration, reported at
`GET /version` as `gateway_mode`:

| `gateway_mode` | When | Keys accepted |
| --- | --- | --- |
| `none` | No gateway configured — mock gateway (full sandbox) | test keys |
| `test` | Stripe `sk_test…` / Razorpay `rzp_test…` keys | test keys |
| `live` | Real gateway keys — real money | live keys |

<Warning>
If a key's mode doesn't match the server, the request is rejected with
`401 key_mode_mismatch`. This is deliberate: a **test key can never run against a
live-money server**, and a live key won't run on a non-live one. Build and test
against a `none`/`test` instance with a test key — see
[Testing & Sandbox](/testing).
</Warning>

## Generate a key

1. Go to **Settings → API Keys** in your dashboard
2. Click **Create API Key** and choose **Test** or **Live**
3. Copy and store it securely — the full key is shown only once

Or via the API — the mode defaults to `test`:

```bash
curl -X POST https://api.recurso.dev/v1/developer/keys \
  -H "Authorization: Bearer $RECURSO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "name": "Server key", "mode": "live" }'
```

## Usage

Pass your API key in the `Authorization` header as a Bearer token:

```bash
curl https://api.recurso.dev/v1/customers \
  -H "Authorization: Bearer $RECURSO_API_KEY"
```

Mutating requests also accept an `X-Idempotency-Key` header so retries never
duplicate work — see [Idempotency](/advanced/idempotency).

## Revoke a Key

If a key is compromised:

1. Go to **Settings → API Keys**
2. Click **Revoke** on the compromised key
3. Generate a new key
4. Update your applications

<Info>
Revoking a key is immediate and permanent.
</Info>
