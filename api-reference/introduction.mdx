## Base URL

```
https://billing.example.com/v1
```

<Note>
`billing.example.com` is a placeholder — substitute the domain where your Recurso instance is deployed. All API examples in this reference use the same placeholder.
</Note>

## Authentication

Authenticate using API keys in the `Authorization` header:

```bash
curl https://billing.example.com/v1/customers \
  -H "Authorization: Bearer rsk_live_xxx"
```

<Warning>
Keep your API keys secret. Never expose them in client-side code.
</Warning>

### API Key Types

| Type | Prefix | Use Case |
|------|--------|----------|
| Live | `rsk_live_` | Production |
| Test | `rsk_test_` | Development/testing |

Test keys only work with test data and test payment methods.

## Request Format

All requests use JSON:

```bash
curl -X POST https://billing.example.com/v1/customers \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Acme Corp",
    "email": "billing@acme.com"
  }'
```

## Response Format

Responses are JSON with consistent structure:

```json
{
  "id": "cust_abc123",
  "object": "customer",
  "name": "Acme Corp",
  "email": "billing@acme.com",
  "created_at": "2024-01-15T10:30:00Z"
}
```

### List Responses

Collections include pagination:

```json
{
  "object": "list",
  "data": [
    { "id": "cust_abc", ... },
    { "id": "cust_def", ... }
  ],
  "has_more": true,
  "total_count": 150
}
```

## Pagination

Use `limit` and `offset` for pagination:

```bash
GET /v1/customers?limit=20&offset=40
```

| Parameter | Default | Max |
|-----------|---------|-----|
| `limit` | 10 | 100 |
| `offset` | 0 | - |

## Errors

Errors include helpful details:

```json
{
  "error": {
    "type": "invalid_request",
    "code": "resource_not_found",
    "message": "Customer cust_xyz not found",
    "param": "customer_id"
  }
}
```

### HTTP Status Codes

| Code | Meaning |
|------|---------|
| `200` | Success |
| `201` | Created |
| `400` | Bad Request |
| `401` | Unauthorized |
| `404` | Not Found |
| `422` | Validation Error |
| `429` | Rate Limited |
| `500` | Server Error |

## Rate Limits

Rate limits depend on your [Recurso plan](/pricing):

| Plan | Requests/min | Description |
|------|--------------|-------------|
| [Open Source](/pricing) | Unlimited (self-configured) | Self-hosted, configure your own limits |
| [Cloud](/pricing) | 600 | Managed hosting at $299/mo |
| [Enterprise](/pricing) | Custom | Dedicated infrastructure with custom limits |

<Tip>
Self-hosted users can configure their own rate limits with no restrictions. See the [Developer Settings guide](/advanced/developer-settings) for configuration options.
</Tip>

Rate limit info in response headers:

```
X-RateLimit-Limit: 600
X-RateLimit-Remaining: 599
X-RateLimit-Reset: 1705312800
```

## Idempotency

For POST requests, use idempotency keys to prevent duplicate operations:

```bash
curl -X POST https://billing.example.com/v1/subscriptions \
  -H "Authorization: Bearer $API_KEY" \
  -H "Idempotency-Key: unique-request-id-123" \
  -d '{ ... }'
```

<Tip>
Use UUIDs or your own unique identifiers as idempotency keys.
</Tip>

## Versioning

API version in header (optional):

```
Recurso-Version: 2024-01-01
```

Without version, you get the latest stable API.
